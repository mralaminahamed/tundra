## 1. Overview

This document specifies the persistence layer of Tundra: the PostgreSQL 18 schema that backs `tundrad`, the Valkey 8 cache and pub/sub topology, the on-disk layout used by the agent and the daemon, and the migration discipline that ties them together. It is the engineering reference for anyone writing SQL, modifying the data model, or reasoning about state ownership inside Tundra.

The implementation plan (`tundra-technical-implementation-plan-v2.md`, §7) gives a high-level table inventory; this document is the long form. Where the implementation plan says "30+ tables," this document enumerates every one of them with rationale.

### 1.1 Conventions

The schema follows a small set of strict conventions enforced by code review:

Every table uses **snake_case** plural names (`sites`, `deployments`). Every primary key is named `id` and is **`uuidv7`** — generated by PostgreSQL 18's native `uuidv7()` function. UUIDv7 sorts lexicographically by creation time, which gives us cheap chronological ordering and good B-tree locality without an additional integer surrogate. Every table carries `created_at timestamptz NOT NULL DEFAULT now()` and `updated_at timestamptz NOT NULL DEFAULT now()`. Soft-deletable tables also carry `deleted_at timestamptz NULL` with an index. The `updated_at` column is maintained by a `BEFORE UPDATE` trigger declared once on a generic function and attached per table.

Foreign keys are named `<singular>_id` (`site_id`, `server_id`). Every foreign key declares `ON DELETE` and `ON UPDATE` behavior explicitly — we do not rely on PostgreSQL's `NO ACTION` default. Composite indexes are named `idx_<table>_<columns>` and listed alongside the table that owns them. Boolean columns are named `is_*` or `has_*` (`is_active`, `has_passkey`). Enumerated values are `text` columns with a `CHECK` constraint (or a PostgreSQL `ENUM` for tightly bounded sets); we prefer `text` + `CHECK` for sets that may evolve, since adding a value to an `ENUM` requires an `ALTER TYPE` that takes a heavy lock.

Money is `numeric(14, 4)` to support fractional units; Tundra's pricing tables (used by the optional billing plugin) are denominated in `numeric(14, 2)` BDT. Time is always `timestamptz`; date-only fields are `date`. We never use `timestamp without time zone` — every timestamp carries its zone offset and is stored as UTC.

JSON is stored as `jsonb`. Long-form text uses `text` (PostgreSQL has no varchar/length advantage). Binary data such as encrypted secret payloads uses `bytea`. Postgres-specific types (`inet`, `cidr`, `macaddr`, `tsvector`, `interval`) are used freely where they express intent better than a generic alternative.

## 2. Storage Topology

Tundra uses three storage tiers, each with explicit ownership:

```
┌─────────────────────────────────────────────────────────────┐
│  PostgreSQL 18 — durable, transactional, source of truth   │
│  All structured records, all state machines, all history   │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  Valkey 8 — ephemeral, fast, derived state                 │
│  Cache, pub/sub, queue (Tundra job queue), rate limits     │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  Disk — large blobs, logs, backup repositories             │
│  /var/lib/tundra/{logs,backups,artifacts,plugins,...}      │
└─────────────────────────────────────────────────────────────┘
```

The **rule of ownership**: PostgreSQL is authoritative. If Valkey crashes and is rebuilt empty, the system recovers — it just runs slower until caches warm. If disk is wiped (logs, build artifacts, backup repositories), users lose the data those files represent, but the panel itself recovers. If PostgreSQL is corrupted without a backup, Tundra's own backup infrastructure (yes, recursively) is the recovery path; see §7.

### 2.1 PostgreSQL Configuration

Targets PostgreSQL 18 with the following extensions enabled in the Tundra database:

```sql
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";          -- legacy uuid_generate_v4 (kept for compat)
CREATE EXTENSION IF NOT EXISTS "pgcrypto";           -- gen_random_bytes, digest(...)
CREATE EXTENSION IF NOT EXISTS "btree_gin";          -- composite GIN indexes for jsonb + status filters
CREATE EXTENSION IF NOT EXISTS "pg_trgm";            -- trigram search for sites/domains by partial name
CREATE EXTENSION IF NOT EXISTS "citext";             -- case-insensitive text (emails, domain names)
```

`uuidv7()` is native in PostgreSQL 18; no extension required. Postgres logical replication is **off** by default (Tundra's HA story uses backup + restore, not replication, in v1.0); the `wal_level` is `replica`, sufficient for streaming backup via `pg_basebackup`.

Tunables enforced at install time (see `tundrad-installer` in the Implementation Plan §17):

| Setting | Value | Rationale |
|---------|-------|-----------|
| `shared_buffers` | 25% of RAM | Standard PG guidance |
| `effective_cache_size` | 70% of RAM | OS page cache + shared_buffers |
| `work_mem` | 16 MB | Conservative; bumped per-query for reports |
| `maintenance_work_mem` | 256 MB | For VACUUM, index builds |
| `max_connections` | 200 | The agent fleet pool sizes against this |
| `wal_compression` | `on` | Free CPU, less WAL volume |
| `random_page_cost` | 1.1 | Assumes SSD; safe default for Tundra deployments |
| `synchronous_commit` | `on` | Tundra's small writes are not bottleneck-y |

### 2.2 Valkey Configuration

Valkey 8 (the open Redis fork) is used with **two databases** (DB 0 and DB 1):

- **DB 0 — Cache & rate limits.** Short-lived keys, fully evictable. `maxmemory-policy allkeys-lru`. Reset on restart is acceptable.
- **DB 1 — Job queue & locks.** Sidekiq-style queue keys, distributed locks. `maxmemory-policy noeviction` — losing queue entries is unacceptable. Persistence via AOF with `everysec` fsync; a Valkey crash loses ≤1 second of enqueues, recovers cleanly.

Pub/sub channels are namespaced as `tundra:events:<topic>` (e.g., `tundra:events:deploys`, `tundra:events:server-metrics`). `tundrad` publishes; the WebSocket gateway subscribes and forwards to connected operators.

### 2.3 On-Disk Layout

```
/var/lib/tundra/
├── data/                          # tundrad runtime data
│   ├── master.key                 # Master encryption key, mode 0400, root only
│   ├── ca/                        # Internal mTLS CA (cert + key for agent fleet)
│   └── jwks/                      # Rotating JWKS for session JWTs
├── logs/
│   ├── tundrad.log                # Daemon log (rotated daily, 14d retention)
│   ├── audit.log                  # Append-only audit log (90d retention; archived)
│   └── deploy/<site_uuid>/<deployment_uuid>.log
├── artifacts/
│   ├── builds/<site_uuid>/<deployment_uuid>/   # Build outputs awaiting promotion
│   └── releases/<site_uuid>/                   # Active and previous releases (rolling 5)
├── backups/
│   ├── repos/<repo_uuid>/         # Restic repositories (the most common backend)
│   └── tmp/                       # Streaming staging area
├── plugins/
│   ├── installed/<plugin_id>/     # Installed plugin bundles (manifest + WASM/native)
│   └── data/<plugin_id>/          # Per-plugin private data dir (capability-scoped)
└── tmp/                           # Scratch space for migrations, imports, exports
```

The agent (on each managed server) has its own simpler tree at `/var/lib/tundra-agent/`:

```
/var/lib/tundra-agent/
├── agent.toml                     # Local config
├── credentials/                   # mTLS client cert + key (rotated by tundrad)
├── state/                         # Local state cache (idempotency keys, run history)
└── logs/agent.log
```

## 3. PostgreSQL Schema — Module-by-Module DDL

The DDL below is the canonical schema. Migrations in `migrations/` are the source of truth; this document captures intent, structure, and the tables that warrant explicit DDL. Smaller satellite tables are summarized in their module's text.

### 3.1 Identity & Access

#### 3.1.1 `operators`

The unified operator (panel user) table.

```sql
CREATE TABLE operators (
  id                    uuid          PRIMARY KEY DEFAULT uuidv7(),
  public_id             text          NOT NULL,                 -- short ID for system-user mapping (8 chars, base32-crockford)
  email                 citext        NOT NULL,
  email_verified_at     timestamptz   NULL,
  full_name             text          NOT NULL,
  role                  text          NOT NULL CHECK (role IN ('owner','admin','operator','readonly')),
  password_hash         text          NULL,                     -- argon2id; nullable if passkey-only
  totp_secret_encrypted bytea         NULL,                     -- AES-256-GCM under master key
  is_active             boolean       NOT NULL DEFAULT true,
  last_login_at         timestamptz   NULL,
  last_login_ip         inet          NULL,
  preferred_locale      text          NOT NULL DEFAULT 'en',
  avatar_path           text          NULL,
  created_at            timestamptz   NOT NULL DEFAULT now(),
  updated_at            timestamptz   NOT NULL DEFAULT now(),
  deleted_at            timestamptz   NULL,

  CONSTRAINT operators_email_unique UNIQUE (email),
  CONSTRAINT operators_public_id_unique UNIQUE (public_id)
);

CREATE INDEX idx_operators_active ON operators (is_active) WHERE deleted_at IS NULL;
CREATE INDEX idx_operators_deleted_at ON operators (deleted_at) WHERE deleted_at IS NOT NULL;

CREATE TRIGGER trg_operators_updated_at
  BEFORE UPDATE ON operators
  FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

The `public_id` is the short identifier used to derive system user names (`tundra-<public_id>`) — this is why operators have it even though `id` is already public-safe; system user names can't be UUIDs.

#### 3.1.2 `sessions`

```sql
CREATE TABLE sessions (
  id                uuid         PRIMARY KEY DEFAULT uuidv7(),
  operator_id       uuid         NOT NULL REFERENCES operators(id) ON DELETE CASCADE,
  refresh_token_hash bytea       NOT NULL,            -- sha256 of the refresh token
  user_agent        text         NULL,
  ip                inet         NULL,
  created_at        timestamptz  NOT NULL DEFAULT now(),
  last_seen_at      timestamptz  NOT NULL DEFAULT now(),
  expires_at        timestamptz  NOT NULL,
  revoked_at        timestamptz  NULL,
  revoke_reason     text         NULL
);

CREATE UNIQUE INDEX idx_sessions_token_hash ON sessions (refresh_token_hash) WHERE revoked_at IS NULL;
CREATE INDEX idx_sessions_operator_active ON sessions (operator_id, expires_at) WHERE revoked_at IS NULL;
```

JWT access tokens are stateless and not stored. Refresh tokens are hashed and stored; rotated on every refresh. Revoked sessions stay in the table for audit purposes and are pruned by a nightly job after 30 days.

#### 3.1.3 `passkeys`

```sql
CREATE TABLE passkeys (
  id              uuid         PRIMARY KEY DEFAULT uuidv7(),
  operator_id     uuid         NOT NULL REFERENCES operators(id) ON DELETE CASCADE,
  credential_id   bytea        NOT NULL,
  public_key      bytea        NOT NULL,
  signature_count bigint       NOT NULL DEFAULT 0,
  transports      text[]       NULL,                  -- ['usb','nfc','ble','internal']
  device_label    text         NULL,
  aaguid          uuid         NULL,
  last_used_at    timestamptz  NULL,
  created_at      timestamptz  NOT NULL DEFAULT now(),
  updated_at      timestamptz  NOT NULL DEFAULT now(),

  CONSTRAINT passkeys_credential_id_unique UNIQUE (credential_id)
);

CREATE INDEX idx_passkeys_operator ON passkeys (operator_id);
```

#### 3.1.4 `roles`, `permissions`, `role_permissions`, `operator_roles`

Standard RBAC quad. Roles are seeded fixtures (`owner`, `admin`, `operator`, `readonly`); permissions are `resource.action` strings (`sites.create`, `servers.read`, `mcp.write:safe`). The `operator_roles` table allows scope-bounded grants — `(operator_id, role_id, scope_type, scope_id)` — useful for "this operator is admin only on these two servers."

```sql
CREATE TABLE roles (
  id          uuid         PRIMARY KEY DEFAULT uuidv7(),
  slug        text         NOT NULL UNIQUE,
  name        text         NOT NULL,
  description text         NULL,
  is_system   boolean      NOT NULL DEFAULT false,    -- system roles cannot be deleted
  created_at  timestamptz  NOT NULL DEFAULT now(),
  updated_at  timestamptz  NOT NULL DEFAULT now()
);

CREATE TABLE permissions (
  id          uuid         PRIMARY KEY DEFAULT uuidv7(),
  slug        text         NOT NULL UNIQUE,            -- 'sites.create', 'mcp.write:safe'
  resource    text         NOT NULL,
  action      text         NOT NULL,
  description text         NULL
);

CREATE INDEX idx_permissions_resource ON permissions (resource);

CREATE TABLE role_permissions (
  role_id       uuid NOT NULL REFERENCES roles(id) ON DELETE CASCADE,
  permission_id uuid NOT NULL REFERENCES permissions(id) ON DELETE CASCADE,
  PRIMARY KEY (role_id, permission_id)
);

CREATE TABLE operator_roles (
  operator_id  uuid NOT NULL REFERENCES operators(id) ON DELETE CASCADE,
  role_id      uuid NOT NULL REFERENCES roles(id) ON DELETE CASCADE,
  scope_type   text NULL,                              -- 'server', 'site', NULL = global
  scope_id     uuid NULL,
  PRIMARY KEY (operator_id, role_id, COALESCE(scope_type,''), COALESCE(scope_id,'00000000-0000-0000-0000-000000000000'::uuid))
);

CREATE INDEX idx_operator_roles_scope ON operator_roles (scope_type, scope_id);
```

#### 3.1.5 `audit_log` (append-only)

```sql
CREATE TABLE audit_log (
  id            uuid         PRIMARY KEY DEFAULT uuidv7(),
  occurred_at   timestamptz  NOT NULL DEFAULT now(),
  actor_type    text         NOT NULL,                  -- 'operator', 'plugin', 'mcp_session', 'system'
  actor_id      uuid         NULL,
  action        text         NOT NULL,                  -- 'site.create', 'plugin.grant_capability'
  resource_type text         NULL,
  resource_id   uuid         NULL,
  ip            inet         NULL,
  user_agent    text         NULL,
  details       jsonb        NOT NULL DEFAULT '{}'::jsonb
);

CREATE INDEX idx_audit_occurred ON audit_log (occurred_at DESC);
CREATE INDEX idx_audit_actor ON audit_log (actor_type, actor_id);
CREATE INDEX idx_audit_resource ON audit_log (resource_type, resource_id);
CREATE INDEX idx_audit_action ON audit_log (action);
```

The `audit_log` is append-only by application convention — there is no `UPDATE` or `DELETE` from any code path. It is the journal that the Security Audit document (companion to this) cross-references. Old rows are exported to compressed Parquet files in `/var/lib/tundra/logs/audit-archive/` and pruned from the table after 90 days.

### 3.2 Servers

The fleet of managed servers, plus the local services Tundra knows how to operate on each.

#### 3.2.1 `servers`

```sql
CREATE TABLE servers (
  id                  uuid          PRIMARY KEY DEFAULT uuidv7(),
  name                text          NOT NULL,
  hostname            text          NOT NULL,
  region              text          NULL,                       -- operator-defined free-text
  public_ip           inet          NULL,
  private_ip          inet          NULL,
  os                  text          NOT NULL,                   -- 'ubuntu-24.04', 'debian-12'
  os_version          text          NOT NULL,
  arch                text          NOT NULL,                   -- 'x86_64', 'aarch64'
  agent_version       text          NULL,
  status              text          NOT NULL DEFAULT 'provisioning'
                                  CHECK (status IN ('provisioning','active','degraded','offline','disabled')),
  agent_last_seen_at  timestamptz   NULL,
  capabilities        jsonb         NOT NULL DEFAULT '{}'::jsonb,
                                                                  -- { "php": ["8.2","8.3","8.4"], "docker": "27.x", ... }
  resources_total     jsonb         NOT NULL DEFAULT '{}'::jsonb, -- { "cpu_cores": 4, "ram_mb": 8192, "disk_gb": 200 }
  agent_cert_fingerprint text       NULL,
  notes               text          NULL,
  created_at          timestamptz   NOT NULL DEFAULT now(),
  updated_at          timestamptz   NOT NULL DEFAULT now(),
  deleted_at          timestamptz   NULL,

  CONSTRAINT servers_hostname_unique UNIQUE (hostname)
);

CREATE INDEX idx_servers_status ON servers (status) WHERE deleted_at IS NULL;
CREATE INDEX idx_servers_last_seen ON servers (agent_last_seen_at);
```

#### 3.2.2 `agent_credentials`

```sql
CREATE TABLE agent_credentials (
  id                uuid         PRIMARY KEY DEFAULT uuidv7(),
  server_id         uuid         NOT NULL REFERENCES servers(id) ON DELETE CASCADE,
  cert_pem          text         NOT NULL,
  cert_fingerprint  text         NOT NULL,
  not_before        timestamptz  NOT NULL,
  not_after         timestamptz  NOT NULL,
  revoked_at        timestamptz  NULL,
  revoke_reason     text         NULL,
  rotated_from_id   uuid         NULL REFERENCES agent_credentials(id) ON DELETE SET NULL,
  created_at        timestamptz  NOT NULL DEFAULT now()
);

CREATE INDEX idx_agent_creds_server ON agent_credentials (server_id) WHERE revoked_at IS NULL;
CREATE UNIQUE INDEX idx_agent_creds_fingerprint ON agent_credentials (cert_fingerprint) WHERE revoked_at IS NULL;
```

The agent's mTLS client cert is short-lived (90 days) and rotated automatically by `tundrad`; the rotation chain is preserved via `rotated_from_id` for forensics.

#### 3.2.3 `services` (managed services on servers)

```sql
CREATE TABLE services (
  id            uuid          PRIMARY KEY DEFAULT uuidv7(),
  server_id     uuid          NOT NULL REFERENCES servers(id) ON DELETE CASCADE,
  kind          text          NOT NULL,                          -- 'nginx', 'php-fpm', 'postgresql', 'valkey', 'systemd-unit'
  name          text          NOT NULL,                          -- service name on the host
  version       text          NULL,
  managed       boolean       NOT NULL DEFAULT true,             -- false = pre-existing, observed but not managed
  status        text          NOT NULL DEFAULT 'unknown'
                                CHECK (status IN ('running','stopped','failed','unknown')),
  config        jsonb         NOT NULL DEFAULT '{}'::jsonb,
  last_check_at timestamptz   NULL,
  created_at    timestamptz   NOT NULL DEFAULT now(),
  updated_at    timestamptz   NOT NULL DEFAULT now(),

  CONSTRAINT services_unique UNIQUE (server_id, kind, name)
);

CREATE INDEX idx_services_server_kind ON services (server_id, kind);
CREATE INDEX idx_services_status ON services (status);
```

#### 3.2.4 Satellite tables (architectural)

- **`packages`** — installed OS packages per server (name, version, source). Refreshed on a 4h cadence; used for security advisory matching.
- **`firewall_rules`** — per-server iptables/nftables rule set declared by the operator and applied by the agent (`direction`, `protocol`, `port_from`, `port_to`, `source_cidr`, `action`, `priority`).
- **`server_metrics_state`** — last-known metrics snapshot per server, denormalized for fast dashboard rendering. Updated by the metrics ingestion path; the historical time series lives in `metrics_samples` (§3.12).

### 3.3 Sites

The unit of work most operators interact with.

#### 3.3.1 `sites`

```sql
CREATE TABLE sites (
  id                uuid          PRIMARY KEY DEFAULT uuidv7(),
  name              text          NOT NULL,
  primary_domain    citext        NOT NULL,
  server_id         uuid          NOT NULL REFERENCES servers(id) ON DELETE RESTRICT,
  application_id    uuid          NULL,                              -- forward ref; set after application created
  status            text          NOT NULL DEFAULT 'provisioning'
                                CHECK (status IN ('provisioning','active','suspended','migrating','archived')),
  document_root     text          NOT NULL,                          -- absolute path on the server
  base_path         text          NULL,                              -- subpath for the app, default '/'
  notes             text          NULL,
  created_at        timestamptz   NOT NULL DEFAULT now(),
  updated_at        timestamptz   NOT NULL DEFAULT now(),
  deleted_at        timestamptz   NULL,

  CONSTRAINT sites_primary_domain_unique UNIQUE (primary_domain)
);

CREATE INDEX idx_sites_server ON sites (server_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_sites_status ON sites (status) WHERE deleted_at IS NULL;
CREATE INDEX idx_sites_primary_domain_trgm ON sites USING gin (primary_domain gin_trgm_ops);
```

#### 3.3.2 `applications`

```sql
CREATE TABLE applications (
  id                uuid          PRIMARY KEY DEFAULT uuidv7(),
  site_id           uuid          NOT NULL REFERENCES sites(id) ON DELETE CASCADE,
  kind              text          NOT NULL                          -- 'static','php','laravel','nodejs','python','go','rust','docker'
                                CHECK (kind IN ('static','php','laravel','nodejs','python','go','rust','docker')),
  runtime_version   text          NOT NULL,                          -- '8.4', '22-lts', '3.13'
  build_command     text          NULL,
  start_command     text          NULL,
  process_count     int           NOT NULL DEFAULT 1,
  health_check_path text          NOT NULL DEFAULT '/',
  source_kind       text          NOT NULL                          -- 'github','gitlab','blank','template'
                                CHECK (source_kind IN ('github','gitlab','blank','template','tarball')),
  source_config     jsonb         NOT NULL DEFAULT '{}'::jsonb,    -- repo, branch, etc.
  current_release_id uuid         NULL,                              -- set after first deploy
  resources_limits  jsonb         NOT NULL DEFAULT '{}'::jsonb,
  created_at        timestamptz   NOT NULL DEFAULT now(),
  updated_at        timestamptz   NOT NULL DEFAULT now(),

  CONSTRAINT applications_site_unique UNIQUE (site_id)
);
```

Sites and applications are 1:1 in v1.0 — every site has exactly one application. The split exists because future versions may support multi-app sites (a Laravel API alongside a Next.js frontend on the same domain), and because keeping site-level concerns (TLS, domain) separate from application-level concerns (runtime, process model) is cleaner.

#### 3.3.3 `deployments`

```sql
CREATE TABLE deployments (
  id              uuid          PRIMARY KEY DEFAULT uuidv7(),
  application_id  uuid          NOT NULL REFERENCES applications(id) ON DELETE CASCADE,
  site_id         uuid          NOT NULL REFERENCES sites(id) ON DELETE CASCADE,
  triggered_by    text          NOT NULL                          -- 'manual','webhook','schedule','rollback','plugin'
                                CHECK (triggered_by IN ('manual','webhook','schedule','rollback','plugin','migration')),
  triggered_by_id uuid          NULL,                              -- operator_id, plugin_id, etc.
  source_ref      text          NULL,                              -- git SHA, tarball hash, etc.
  source_message  text          NULL,                              -- commit message
  status          text          NOT NULL DEFAULT 'queued'
                                CHECK (status IN ('queued','running','succeeded','failed','cancelled')),
  started_at      timestamptz   NULL,
  finished_at     timestamptz   NULL,
  log_path        text          NULL,                              -- path to the build log on disk
  error           text          NULL,
  release_path    text          NULL,                              -- path to the released artifact
  metadata        jsonb         NOT NULL DEFAULT '{}'::jsonb,
  created_at      timestamptz   NOT NULL DEFAULT now(),
  updated_at      timestamptz   NOT NULL DEFAULT now()
);

CREATE INDEX idx_deployments_app_created ON deployments (application_id, created_at DESC);
CREATE INDEX idx_deployments_site_created ON deployments (site_id, created_at DESC);
CREATE INDEX idx_deployments_status ON deployments (status) WHERE status IN ('queued','running');
```

The `deployments` table is one of the hottest in the system; the partial index on `(status)` for queued/running rows keeps the operator dashboard fast even with a million historical deployments.

#### 3.3.4 Satellite tables (architectural)

- **`env_vars`** — `(application_id, key, value_encrypted, is_secret, updated_by, updated_at)`. Plaintext values are stored only when `is_secret = false`; otherwise the value is AES-256-GCM encrypted under the master key.
- **`scheduled_tasks`** — cron-style scheduled tasks attached to applications. Status flag for enable/disable; last-run and last-exit recorded.
- **`releases`** — physical release artifacts on disk. Older releases are pruned to last 5 by default.
- **`site_aliases`** — secondary domains pointing at the same site. Each alias is independently tracked for TLS issuance.
- **`site_health_checks`** — declarative health probes (path, interval, expected status code, timeout). Results inform `sites.status`.

### 3.4 Domains

#### 3.4.1 `domains`

```sql
CREATE TABLE domains (
  id                  uuid          PRIMARY KEY DEFAULT uuidv7(),
  apex                citext        NOT NULL,                       -- 'example.com'
  registrar_plugin_id uuid          NULL,                            -- which plugin manages registration (Namecheap, etc.)
  dns_managed_by      text          NOT NULL DEFAULT 'tundra'       -- 'tundra' | 'external' | 'registrar'
                                  CHECK (dns_managed_by IN ('tundra','external','registrar')),
  registration_expires_at timestamptz NULL,
  auto_renew          boolean       NOT NULL DEFAULT true,
  ns_locked           boolean       NOT NULL DEFAULT false,
  notes               text          NULL,
  created_at          timestamptz   NOT NULL DEFAULT now(),
  updated_at          timestamptz   NOT NULL DEFAULT now(),

  CONSTRAINT domains_apex_unique UNIQUE (apex)
);

CREATE INDEX idx_domains_dns_mgmt ON domains (dns_managed_by);
CREATE INDEX idx_domains_expiry ON domains (registration_expires_at) WHERE auto_renew = true;
```

#### 3.4.2 `dns_records`

```sql
CREATE TABLE dns_records (
  id            uuid         PRIMARY KEY DEFAULT uuidv7(),
```

**Partitioning.** `metrics_samples` is **partitioned by `occurred_at`** (RANGE) into daily partitions, with automatic creation of the next 7 days and detachment of partitions older than 90 days (default; configurable). Detached partitions are exported to Parquet and dropped. The partition discipline is configured via the `pg_partman` extension at install time.

```sql
-- Example partition declaration

-- pg_partman handles creation of:
-- metrics_samples_p2026_05_02
-- metrics_samples_p2026_05_03
-- ...
```

This keeps the index size and vacuum cost predictable even with millions of samples per day.

## 4. Indexing Strategy

The schema declares its hot-path indexes inline. Beyond those, the following secondary indexes are added in a dedicated migration with explicit rationale:

| Index | Why |
|-------|-----|
| `idx_sites_search` — GIN trigram on `(primary_domain || ' ' || name)` | The global command palette and site search bar; sub-50ms partial match for 10k+ sites |
| `idx_deployments_site_succeeded_recent` — partial index on `(site_id, created_at DESC) WHERE status = 'succeeded'` | "When was the last successful deploy?" called on every site detail page |
| `idx_audit_log_actor_recent` — partial on `(actor_id, occurred_at DESC) WHERE actor_type = 'operator'` | The "your recent activity" view in the operator menu |
| `idx_jobs_pending` — partial on `(kind, next_run_at) WHERE status = 'pending'` | The job dispatcher's hot path |
| `idx_metrics_scope_metric_recent` — covering on `(scope_type, scope_id, metric)` INCLUDE `(value, occurred_at)` for last 24h partition | Dashboard sparklines |

We do **not** add indexes speculatively. Every index is justified by a measured query plan from the test fleet, captured in the migration commit message.

## 5. Constraints & Data Integrity

The schema relies heavily on database-level constraints rather than application-level guards, by deliberate principle:

**Foreign keys are mandatory.** Every referential relationship has a foreign key with declared `ON DELETE` and `ON UPDATE` behavior. The matrix:

| When you delete | Cascades to | Restricts on |
|-----------------|-------------|--------------|
| `operator` | `sessions`, `passkeys`, `operator_roles`, `mcp_sessions` | `audit_log` (NULL'd via SET NULL); `plugins.installed_by` (SET NULL); `deployments.triggered_by_id` (SET NULL) |
| `server` | `services`, `agent_credentials`, `packages`, `firewall_rules`, `database_servers` | `sites` (RESTRICT — must be moved or deleted first) |
| `site` | `applications` (which cascades to `deployments`, etc.), `certificates`, `site_aliases`, `site_health_checks` | — |
| `application` | `deployments`, `env_vars`, `scheduled_tasks`, `releases` | `databases` (SET NULL — preserves the DB even if the app is removed) |
| `domain` | `dns_records`, `dns_zones`, `domain_registrations`, `ns_history` | `sites.primary_domain` (no FK; managed at the application level for flexibility) |
| `plugin` | `plugin_capabilities`, `plugin_settings`, `plugin_jobs`, `plugin_events`, `plugin_kv` | `audit_log` rows referencing the plugin (logical only — no FK from audit_log) |
| `database_server` | `databases`, `db_users` | — |

**CHECK constraints** enforce enumerated value domains. The application's enum types match these constraints exactly; updating an enum requires updating the constraint and writing a corresponding migration.

**`UNIQUE` constraints** enforce real-world uniqueness (one site per primary domain, one mailbox per local-part within a domain, etc.). They are duplicated in application-level Zod schemas for nicer error messages, but the database is authoritative.

**`NOT NULL`** is applied wherever absence has no semantic meaning. `nullable` columns carry an inline comment in the migration explaining the meaning of `NULL`.

Generated columns are used for computed scalars that must stay in sync with their inputs:

```sql
ALTER TABLE fee_invoices ADD COLUMN balance_amount numeric(12,2)
  GENERATED ALWAYS AS (total_amount - paid_amount) STORED;
```

This eliminates the entire class of bugs where a denormalized field drifts from its source.

## 6. Soft Deletes

Soft deletes apply to a deliberate subset of tables — those where audit, restoration, or referential integrity make hard deletion unsafe:

- `operators` — for audit reasons; an operator may leave but their actions in `audit_log` should still resolve.
- `sites` — for "archive then delete" UX and to allow restore of a recently-archived site.
- `servers` — same reasoning.
- `applications` — implied by `sites` deletion cascade.

Other tables are hard-deleted. `dns_records`, `env_vars`, `mailboxes` (when an operator says delete, they mean delete) — but their lifecycles are also reflected in `audit_log`, so the historical record persists independent of the row's existence.

Soft-delete queries always include `WHERE deleted_at IS NULL`. The Rust query layer (sqlx) uses a small wrapper `Soft<Table>` that injects this filter automatically; explicit `.with_deleted()` is required to bypass it, and only a few admin paths do so.

## 7. Backup & Restore — The Recursive Question

Tundra backs up its operators' workloads. Tundra also needs to back up itself. The two paths are deliberately separate:

**Self-backup** is performed by a small, opinionated script (`tundra-self-backup`) that runs as a systemd timer on the control-plane host. It produces:

1. A `pg_dump --format=custom` of the Tundra Postgres database, encrypted with the operator's GPG public key (configured at install time).
2. A tar of `/var/lib/tundra/data/` (which contains the master key, mTLS CA, JWKS).
3. Checksums and a small manifest describing what's in the bundle.

The bundle is uploaded to a backup target the operator pre-configures **separately from the primary `backup_targets` table** — typically a different cloud account, a different region, or a separate physical disk. The principle is that a backup of Tundra can never live only in storage that Tundra itself manages.

**Restore** is performed by `tundra-restore <bundle.tar.gpg>`, which:

1. Halts `tundrad` if running.
2. Decrypts the bundle and verifies checksums.
3. Re-creates the Postgres role and database (idempotent).
4. Restores the dump.
5. Restores `/var/lib/tundra/data/` with strict permission preservation.
6. Re-imports the master key, verifies it can decrypt one known secret, exits non-zero on failure.
7. Starts `tundrad`.

Documented end to end in the Deployment Runbook (companion document).

## 8. Migration Discipline

Schema migrations are managed by `sqlx-cli`. Migrations are immutable once merged to `main`; modifications happen as new migrations. Migration files live in `migrations/` with the naming convention `<timestamp>_<verb>_<noun>.sql` (`20260502120000_create_sites.sql`).

**Up-only.** We do not write `down` migrations. Reverting a deploy is a code revert + new migration, not a database rollback. This eliminates an entire class of "the down migration was never tested" bugs and matches the production reality: nobody runs a down migration against a database with hours-old data.

**Reviewable.** Every migration is reviewed for:

- Locking behavior. Does this migration block writes? `CREATE INDEX CONCURRENTLY` is mandatory for indexes on tables larger than 10k rows.
- Backfill cost. Migrations that add a NOT NULL column with a default must use `ADD COLUMN ... DEFAULT ... NOT NULL` which Postgres 11+ handles without a rewrite. Older patterns are forbidden.
- Reversibility of the *application* path. If we ship the migration, can we still ship the new code an hour later? The answer must always be yes.
- Constraint validation. New CHECK constraints on populated tables use `NOT VALID` then `VALIDATE CONSTRAINT` to avoid blocking writes.

**Tested.** Every migration runs in CI against a fresh database and a database loaded from a representative production snapshot (the test fleet's state). Migrations that take longer than 30 seconds in either context are flagged for review; migrations that take longer than 5 minutes against the production-snapshot are gated behind an explicit `slow_migration: true` flag in the CI config and announced in release notes.

## 9. Encryption at Rest

PostgreSQL is configured for transparent disk encryption via the host (LUKS on Linux). Tundra adds application-level encryption for sensitive fields:

- **Master key** lives at `/var/lib/tundra/data/master.key` (32 bytes, mode 0400, owned by the `tundra` system user). It is generated at install time; backup and restore is the operator's responsibility.
- **Encrypted columns** use AES-256-GCM with the master key as the KEK. The data key is derived per-column-family via HKDF (`info = column_family_name`).
- The Rust code path is centralized in `tundra::crypto::EncryptedField<T>` — a sqlx custom type that handles encrypt-on-write and decrypt-on-read transparently. Every encrypted column maps to this type. There is no path where a developer can write plaintext into an encrypted column by accident.

**Columns currently encrypted** (every `*_encrypted bytea NOT NULL` in the DDL above): `operators.totp_secret_encrypted`, `passkeys.public_key` (yes, even though public — for the same encrypt-everywhere consistency), `db_users.password_encrypted`, `database_servers.superuser_password_encrypted`, `dkim_keys.private_key_encrypted`, `acme_accounts.account_key_encrypted`, `certificates.key_encrypted`, plus `env_vars.value_encrypted` when `is_secret = true`.

Rotation of the master key is supported: `tundra master-key rotate` re-encrypts every encrypted column under a new key, in chunks, online, with checksums verified before swap. Documented in the Deployment Runbook.
